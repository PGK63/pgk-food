# Руководство пользователя — роль CURATOR (Куратор)

## Роль и доступ

**Что доступно в роли `CURATOR`:**
1. `dashboard` — главный экран куратора с дедлайн-уведомлением.
2. `roster` — табель питания по студентам.
3. `stats` — статистика по группе за дату.
4. `categories` — назначение категории студентам.
5. `reports` — отчеты по своим группам.
6. `settings` — настройки (подсказки, масштаб, уведомления).

**Важно:**
1. Основной рабочий порядок: `Табель` -> `Категории` (при необходимости) -> `Статистика`/`Отчеты`.
2. Для корректного сохранения табеля у студентов должна быть назначена категория.

## Быстрый старт

### Сценарий LGN-01-CUR. Войти и открыть табель

**Цель:** перейти к ежедневной работе куратора.

**Перед началом:**
1. Подготовьте логин и пароль.

**Шаги:**
1. Войдите в приложение.
2. Нажмите `Табель`.
3. Проверьте выбранную группу и дату.

**Ожидаемый результат:**
1. Открыт экран табеля с карточками студентов.

**Если что-то пошло не так:**
1. Если не загружается список групп, повторите позже при стабильной сети.
2. Когда группа не выбрана, выберите доступную группу в выпадающем списке.
3. После сообщения об ошибке табеля повторите загрузку экрана.

**Автологика системы:**
1. При первом открытии загружается список ваших групп и выбирается первая доступная.
2. По умолчанию выбирается следующая учебная неделя.

## Основные сценарии

### Сценарий CUR-01. Открыть табель по дате

**Цель:** выбрать рабочую дату табеля и увидеть состояние по студентам.

**Перед началом:**
1. Откройте `Табель`.

**Шаги:**
1. Нажмите на плашку даты.
2. Выберите нужный день в календаре.
3. Дождитесь загрузки карточек студентов.
4. При необходимости смените группу в селекторе `Группа`.

**Ожидаемый результат:**
1. Табель показан для выбранной даты и группы.

**Если что-то пошло не так:**
1. Если дата не выбирается, проверьте ограничения по доступности дат.
2. Когда по дате пусто, убедитесь, что выбран правильный диапазон и группа.
3. После сетевой ошибки повторите загрузку.

**Автологика системы:**
1. Доступны только будние дни (Пн–Пт).
2. В обычном режиме читаемы даты начиная со следующей недели.
3. В тестовом режиме доступны даты начиная с текущей недели.

---

### Сценарий CUR-02. Переключить группу и проверить состав

**Цель:** работать с нужной группой без выхода из экрана.

**Перед началом:**
1. В аккаунте должно быть более одной группы.

**Шаги:**
1. В табеле откройте выпадающий список `Группа`.
2. Выберите нужную группу.
3. Дождитесь обновления карточек студентов.
4. Проверьте, что отображается корректный состав.

**Ожидаемый результат:**
1. В табеле показаны студенты выбранной группы.

**Если что-то пошло не так:**
1. Если список пуст, проверьте, назначены ли студенты в выбранной группе.
2. Когда группа не меняется, закройте и откройте экран заново.

**Автологика системы:**
1. При смене группы включается отдельный snapshot состояния по экрану куратора.

### Сценарий CUR-03. Проставить питание и причины непитания

**Цель:** заполнить табель по студенту на выбранный день.

**Перед началом:**
1. Откройте карточку нужного студента в табеле.

**Шаги:**
1. Для отметки питания нажмите чип `Завтрак` и/или `Обед`.
2. Если студент не питается, снимите оба чипа.
3. Выберите `Причина непитания` (`Отчислен`, `Больничный`, `Иное`).
4. Для `Больничный`/`Иное` задайте даты `С` и `По`.
5. Для `Иное` заполните `Текст причины`.
6. При необходимости заполните `Комментарий`.

**Ожидаемый результат:**
1. Карточка студента содержит корректные отметки на выбранную дату.

**Если что-то пошло не так:**
1. Если редактирование заблокировано, проверьте, не наступил ли режим `только чтение`.
2. Когда студент отчислен (заморожен), редактирование недоступно по статусу.
3. После выбора периода убедитесь, что дата `По` не раньше даты `С`.

**Автологика системы:**
1. Для категории `Многодетные` разрешен только один прием пищи в день.
2. Для причин `Больничный`/`Иное` даты отсутствия подставляются автоматически текущим днем, если пусты.

### Сценарий CUR-04. Скопировать отметки дня на другую дату

**Цель:** ускорить заполнение повторяющихся дней.

**Перед началом:**
1. В табеле уже есть заполненные отметки на исходную дату.

**Шаги:**
1. Нажмите иконку `Скопировать день`.
2. Выберите целевую дату копирования.
3. Нажмите `Копировать`.
4. Дождитесь завершения операции.

**Ожидаемый результат:**
1. Отметки активных студентов скопированы на целевую дату.

**Если что-то пошло не так:**
1. Если видите `Нет активных студентов для копирования`, проверьте статус студентов в группе.
2. Когда появляется ошибка `STUDENT_CATEGORY_REQUIRED`, откройте `Категории`, назначьте категорию и повторите.
3. После частичной ошибки повторите копирование и проверьте выборочно карточки.

**Автологика системы:**
1. Копирование выполняется только для активных (не замороженных) студентов.
2. Целевая дата выбирается из ближайших редактируемых дат.

### Сценарий CUR-05. Найти студента в табеле

**Цель:** быстро перейти к нужной карточке студента.

**Перед началом:**
1. Откройте `Табель`.

**Шаги:**
1. В поле `Начните искать` введите часть ФИО.
2. Выберите нужного студента в отфильтрованном списке.
3. Внесите изменения в карточку.

**Ожидаемый результат:**
1. Список сужается по введенному тексту.

**Если что-то пошло не так:**
1. Если студент не найден, очистите поиск и проверьте выбранную группу.
2. Когда поиск пустой при корректном ФИО, проверьте регистр/опечатки.

**Автологика системы:**
1. Поиск в табеле выполняется по `fullName` без учета регистра.

### Сценарий CUR-06. Сохранить табель и проверить консистентность

**Цель:** зафиксировать отметки на сервере.

**Перед началом:**
1. Заполните карточки студентов на выбранную дату.

**Шаги:**
1. Нажмите `Сохранить` внизу экрана.
2. Дождитесь завершения сохранения.
3. Перезагрузите экран (смените и верните дату) для контрольной проверки.

**Ожидаемый результат:**
1. Данные сохранены (`Сохранено`).
2. Повторная загрузка отображает те же значения.

**Если что-то пошло не так:**
1. Если видите `Период отсутствия заполнен неверно`, исправьте даты и повторите.
2. Когда видите `Нет активных студентов для сохранения`, проверьте статус студентов.
3. После `STUDENT_CATEGORY_REQUIRED` перейдите в `Категории`, назначьте значения и сохраните снова.

**Автологика системы:**
1. Сохранение выполняется по студентам последовательно.
2. При ошибке категории открывается маршрут в `Категории`.

### Сценарий CUR-07. Работать с дедлайном и режимом «только чтение»

**Цель:** не потерять изменения из-за закрытого окна редактирования.

**Перед началом:**
1. Откройте `Табель`.

**Шаги:**
1. Проверьте, активна ли кнопка `Сохранить`.
2. Если кнопка неактивна, прочитайте предупреждение `Только чтение...`.
3. Переключитесь на доступную редактируемую дату.
4. Выполните сохранение в разрешенном окне.

**Ожидаемый результат:**
1. На закрытой дате табель доступен только для просмотра.
2. На открытой дате сохраняется корректно.

**Если что-то пошло не так:**
1. Если дата не редактируется после пятницы 12:00, переключитесь на следующую допустимую неделю.
2. Когда поведение не совпадает с ожиданиями, уточните, не включен ли тестовый режим.

**Автологика системы:**
1. Для следующей недели действует дедлайн: после пятницы 12:00 эта неделя становится read-only.
2. В тестовом режиме правила смягчены по окну чтения/редактирования.

### Сценарий CUR-08. Подтвердить отчисление в табеле

**Цель:** корректно применить причину `Отчислен`.

**Перед началом:**
1. Для одного или нескольких студентов установлена причина `Отчислен` на выбранную дату.

**Шаги:**
1. Нажмите `Сохранить`.
2. В диалоге подтверждения отчисления нажмите `Подтвердить`.

**Ожидаемый результат:**
1. Табель сохранен с отметками отчисления.
2. Пользователь переходит в замороженный статус и недоступен для дальнейшего назначения питания.

**Если что-то пошло не так:**
1. Если нажали `Отмена`, данные не сохранятся — повторите сохранение при готовности.
2. Когда после сохранения студент не изменил статус, выполните повторную загрузку данных.

**Автологика системы:**
1. Для причины `Отчислен` перед сохранением показывается дополнительное подтверждение.

### Сценарий CAT-01. Назначить категории студентам

**Цель:** устранить ошибки табеля, связанные с отсутствием категории.

**Перед началом:**
1. Откройте экран `Категории`.

**Шаги:**
1. Выберите группу в поле поиска группы.
2. Найдите студента в списке.
3. Нажмите нужную категорию (`СВО` или `Многодетные`).
4. Дождитесь сообщения `Категория обновлена`.

**Ожидаемый результат:**
1. Категория студента сохранена.

**Если что-то пошло не так:**
1. Если студент имеет статус `Отчислен (заморожен)`, категория не редактируется.
2. Когда видите `Ошибка обновления категории`, повторите позже.
3. После массовых изменений вернитесь в `Табель` и выполните повторное сохранение.

**Автологика системы:**
1. Для студентов без категории в списке выводится явный индикатор `Категория не назначена`.

### Сценарий CUR-STAT-01. Проверить статистику группы за дату

**Цель:** сверить факт питания по студентам.

**Перед началом:**
1. Откройте `Статистика`.

**Шаги:**
1. Выберите группу (если доступно несколько).
2. Нажмите на плашку даты и выберите день.
3. Проверьте карточки студентов и бейджи `Завтрак`/`Обед`.

**Ожидаемый результат:**
1. Отображен список статусов питания на выбранную дату.

**Если что-то пошло не так:**
1. Если выводится `Нет данных за эту дату`, проверьте, сохранен ли табель на этот день.
2. Когда группа не выбрана, выберите ее в селекторе.

**Автологика системы:**
1. Выходные даты автоматически нормализуются к ближайшему рабочему дню (пятнице).

### Сценарий CUR-REP-01. Сформировать отчет по своим группам

**Цель:** получить детализацию и сводку по питанию за период.

**Перед началом:**
1. Откройте экран `Отчеты`.

**Шаги:**
1. Выберите `С даты` и `По дату`.
2. При необходимости выберите группу (`Все мои группы` или конкретную).
3. Нажмите `Сформировать`.
4. Проверьте сводный блок и детализацию по строкам.
5. При необходимости выполните `Экспорт CSV` или `Экспорт PDF`.

**Ожидаемый результат:**
1. Загружены данные отчета по выбранному периоду и доступным группам куратора.

**Если что-то пошло не так:**
1. Если дата окончания меньше даты начала, исправьте диапазон.
2. Когда выбрана будущая дата, уменьшите период до текущей даты.
3. После ошибки экспорта повторите при стабильной сети.

**Автологика системы:**
1. В роли куратора вкладка `Подозрения` скрыта.
2. Область данных ограничена группами, закрепленными за куратором.

## Ошибки и нестандартные состояния

1. `После пятницы 12:00 ... только для просмотра` — выбран read-only период.
2. `Период отсутствия заполнен неверно` — дата `По` раньше даты `С`.
3. `STUDENT_CATEGORY_REQUIRED`/пользовательское сообщение о категории — перейдите в `Категории`.
4. `Нет активных студентов для сохранения/копирования` — все пользователи в статусе заморозки или список пуст.
5. `Вы не привязаны ни к одной группе` — обратитесь к регистратору/администратору для назначения группы.
6. `Нет данных за эту дату` в статистике — табель на дату не сохранен или пуст.

## Масштаб интерфейса

### Сценарий UI-SCALE. Изменить масштаб

**Цель:** увеличить или уменьшить размер элементов интерфейса.

**Перед началом:**
1. Откройте `Настройки` (иконка шестеренки на `dashboard`).

**Шаги:**
1. Найдите блок `Масштаб интерфейса`.
2. Передвиньте слайдер до нужного значения.
3. Проверьте предпросмотр в блоке `Пример интерфейса`.
4. Нажмите `Применить`.
5. При необходимости нажмите `Сбросить по умолчанию`.

**Ожидаемый результат:**
1. Масштаб интерфейса изменен и сохранен.

**Если что-то пошло не так:**
1. Если интерфейс стал неудобным, нажмите `Сбросить по умолчанию`.
2. Когда изменения не видны, проверьте, что нажали именно `Применить`.

**Автологика системы:**
1. Допустимый диапазон масштаба: `85%–115%`.
2. По умолчанию: `100%`.
3. Масштаб влияет и на плотность UI, и на размер шрифтов, сохраняется между запусками.

## Уведомления

### Сценарий PUSH. Включить уведомления в приложении

**Цель:** получать push-уведомления по событиям системы.

**Перед началом:**
1. Откройте `Настройки`.

**Шаги:**
1. Найдите блок `Уведомления`.
2. Переключите тумблер `Включить уведомления` в `ON`.
3. На Android 13+ подтвердите системное разрешение, если появится диалог.
4. Дождитесь завершения обновления (индикатор загрузки под тумблером).

**Ожидаемый результат:**
1. Серверная настройка push сохранена.
2. На Android системное разрешение также включено.

**Если что-то пошло не так:**
1. Если видите `Включите системные уведомления в настройках устройства`, откройте настройки ОС и включите уведомления для приложения.
2. Когда видите `Системные уведомления выключены...`, повторите выдачу системного разрешения.
3. После ошибки `Не удалось обновить уведомления` повторите попытку при стабильной сети.

**Автологика системы:**
1. Приложение может один раз автоматически запросить системное разрешение для вошедшего пользователя.

## Подсказки

### Сценарий HINT. Управлять подсказками

**Цель:** включить/выключить обучающие подсказки.

**Перед началом:**
1. Откройте `Настройки`.

**Шаги:**
1. Найдите блок `Подсказки`.
2. Переключите тумблер `Показывать подсказки` в нужное состояние.
3. Откройте нужный экран и при необходимости закройте карточку подсказки через кнопку скрытия.
4. Для полного возврата подсказок включите тумблер снова.

**Ожидаемый результат:**
1. Подсказки отображаются или скрыты в соответствии с настройкой.

**Если что-то пошло не так:**
1. Если подсказка не возвращается на одном экране, включите глобальный тумблер повторно.
2. Когда подсказки мешают, отключите их глобально в настройках.

**Автологика системы:**
1. Без явной ручной настройки подсказки показываются ограниченное время после первого входа (окно 72 часа).
2. Скрытие подсказки на конкретном экране сохраняется отдельно.
3. Повторное глобальное включение очищает скрытия по экранам.

## Автологика системы (почему так)

1. На `dashboard` показывается дедлайн-карточка, если сервер прислал необходимость напоминания или причину блокировки.
2. В табеле есть авто-переход на первую незаполненную дату в пределах рабочей недели (однократно для контекста).
3. Сохранение и копирование табеля прерывается при бизнес-ошибке категории и направляет пользователя в `Категории`.
4. Замороженные (`Отчислен`) студенты отображаются, но не редактируются.
